**Shibuya**는 Hard 난이도의 Windows AD 머신으로, 초기에는 SMB와 Kerberos 사용자 열거를 통해 머신 계정 자격증명을 발견하면서 시작된다. 이후 `svc_autojoin` 계정의 description에서 평문 비밀번호를 획득해 추가 접근 권한을 얻는다.

해당 계정으로 내부를 탐색하던 중 `Windows Imaging Format (.wim)` 파일에서 `simon.watson` 의 NTLM 해시를 발견하고 SSH 접근을 확보한다. 이후 활성 세션을 대상으로 **Cross Session Relay** 공격을 수행해 `nigel.mills` 계정의 해시를 탈취 및 크랙한다.

`nigel.mills` 는 `t1_admin` 그룹 소속으로 AD CS 인증서 템플릿에 대한 권한을 가지고 있으며, 이를 통해 **ESC1 취약점을 악용하여 인증서 기반 권한 상승**을 수행한다. 최종적으로 관리자 권한을 획득하게 된다.

# Vulnerability Analysis

## Initial Enumeration

우선 `null session` 을 이용하여 SMB 인증을 시도하였다. 익명 로그인 자체는 가능하였으나, `share enumeration` 단계에서는 접근이 거부되었다:

```bash
$ nxc smb 10.129.234.42 -u '' -p '' --shares          

SMB         10.129.234.42   445    AWSJPDC0522      [*] Windows Server 2022 Build 20348 x64 (name:AWSJPDC0522) (domain:shibuya.vl) (signing:True) (SMBv1:False)                                                                                                                           
SMB         10.129.234.42   445    AWSJPDC0522      [+] shibuya.vl\: 
SMB         10.129.234.42   445    AWSJPDC0522      [-] Error enumerating shares: STATUS_ACCESS_DENIED
```

이후, [kerbrute](https://github.com/ropnop/kerbrute) 와 SecLists의 사용자 목록을 이용하여 Kerberos 기반 사용자 열거를 수행하였다:

```bash
$ kerbrute userenum /usr/share/seclists/Usernames/xato-net-10-million-usernames.txt -d shibuya.vl --dc shibuya.vl

2026/05/08 05:07:49 >  Using KDC(s):
2026/05/08 05:07:49 >   shibuya.vl:88

2026/05/08 05:07:51 >  [+] VALID USERNAME:       purple@shibuya.vl
2026/05/08 05:07:55 >  [+] VALID USERNAME:       red@shibuya.vl
```

그 결과 `purple`, `red` 두 개의 유효한 계정을 확인할 수 있었다.

이후 발견한 사용자명을 기반으로 Password Spraying을 시도하였으나, 일반 사용자 인증은 모두 실패하였다:

```bash
$ nxc smb 10.129.234.42 -u user.txt -p user.txt --no-bruteforce

SMB         10.129.234.42   445    AWSJPDC0522      [*] Windows Server 2022 Build 20348 x64 (name:AWSJPDC0522) (domain:shibuya.vl) (signing:True) (SMBv1:False)                                                                                                                           
SMB         10.129.234.42   445    AWSJPDC0522      [-] shibuya.vl\purple:purple STATUS_LOGON_FAILURE 
SMB         10.129.234.42   445    AWSJPDC0522      [-] shibuya.vl\red:red STATUS_LOGON_FAILURE
```


하지만 머신 계정일 가능성을 고려하여 `$` 를 붙여 재시도하였다:

```bash
$ nxc smb 10.129.234.42 -u computer.txt -p user.txt --no-bruteforce

SMB         10.129.234.42   445    AWSJPDC0522      [*] Windows Server 2022 Build 20348 x64 (name:AWSJPDC0522) (domain:shibuya.vl) (signing:True) (SMBv1:False)                                                                                                                           
SMB         10.129.234.42   445    AWSJPDC0522      [-] shibuya.vl\PURPLE$:purple STATUS_NOLOGON_WORKSTATION_TRUST_ACCOUNT 
SMB         10.129.234.42   445    AWSJPDC0522      [-] shibuya.vl\RED$:red STATUS_NOLOGON_WORKSTATION_TRUST_ACCOUNT
```

그 결과 `STATUS_NOLOGON_WORKSTATION_TRUST_ACCOUNT` 오류가 반환되었으며, 이를 통해 `PURPLE$`, `RED$` 계정이 실제 컴퓨터 계정임을 확인할 수 있었다.

이후 Kerberos 인증을 사용하기 위해 시간 동기화(`ntpdate`)를 수행한 뒤 `-k` 옵션을 이용하여 재시도한 결과 정상적으로 인증에 성공하였다:

```bash
$ nxc smb 10.129.234.42 -u computer.txt -p user.txt --no-bruteforce --continue-on-success -k

SMB         10.129.234.42   445    AWSJPDC0522      [*] Windows Server 2022 Build 20348 x64 (name:AWSJPDC0522) (domain:shibuya.vl) (signing:True) (SMBv1:False)                                                                                                                           
SMB         10.129.234.42   445    AWSJPDC0522      [+] shibuya.vl\PURPLE$:purple 
SMB         10.129.234.42   445    AWSJPDC0522      [+] shibuya.vl\RED$:red
```

### Registry Hive Extraction

[wimlib](https://wimlib.net/)를 이용해 `AWSJPWK0222-02.wim` 파일 내부를 확인한 결과, `SAM`, `SYSTEM`, `SECURITY` 파일들이 포함되어 있는 것을 확인할 수 있었다:

```bash
$ wimlib-imagex dir AWSJPWK0222-02.wim 1                                                         

# SKIP

/SAM
/SAM.LOG1
/SAM.LOG2
/SECURITY
/SECURITY.LOG1
/SECURITY.LOG2
/SOFTWARE
/SOFTWARE.LOG1
/SOFTWARE.LOG2
/SYSTEM
/SYSTEM.LOG1
/SYSTEM.LOG2

# SKIP
```

이후 `hives` 디렉터리를 생성한 뒤 위 파일들을 로컬 환경으로 추출하였다:

```bash
$ wimlib-imagex extract AWSJPWK0222-02.wim 1 SAM SYSTEM SECURITY --dest-dir hives

[WARNING] Ignoring Windows NT security descriptors of 3 files
Extracting file data: 16 MiB of 16 MiB (100%) done
Done extracting files.
```

`hives` 디렉터리 내부를 확인하면 `Registry Hive` 파일들이 정상적으로 추출된 것을 확인할 수 있다:

### Cross Session Relay

위 관계를 통해 현재 `nigel.mills` 사용자의 로그온 세션이 `AWSJPDC0522` 호스트에 존재함을 확인할 수 있었다.

이 조건에서 [RemotePotato0](https://github.com/antonioCoco/RemotePotato0) 를 활용한 **Cross Session Relay** 공격을 시도할 수 있다.

해당 도구는 동일 호스트에 로그인 중인 다른 사용자 세션을 대상으로 NTLM 인증을 유도할 수 있는 도구이다.

우선 `query session` 명령을 실행해보면 현재 세션 정보가 정상적으로 출력되지 않는 것을 확인할 수 있다:

```text

```

현재 확보한 셸은 SSH를 통해 연결된 제한적인 세션이며, 일반적인 RDP 또는 콘솔 기반의 `interactive logon` 세션과는 차이가 존재한다.

따라서 현재 셸에서는 `query session` 과 같은 명령을 통해 터미널 서비스 세션 정보를 정상적으로 조회하지 못하는 상태였다.

이를 우회하기 위해 [RunasCs](https://github.com/antonioCoco/RunasCs)를 사용하였다.

`RunasCs` 의 `-l 9` 옵션은 **Logon Type 9(`NewCredentials`)** 방식으로 새로운 로그온 컨텍스트를 생성한다. 

이를 통해 기존 SSH 셸과는 다른 컨텍스트에서 `query session` 명령을 실행할 수 있었다.

`Logon Type 9(NewCredentials)` 는 로컬에서 자격증명을 즉시 검증하기보다는 새로운 로그온 컨텍스트를 생성하는 특징이 있다.  

따라서 이 환경에서는 실제로 존재하지 않는 자격증명을 사용하더라도 새로운 컨텍스트에서 프로세스를 실행할 수 있었으며, 이를 통해 세션 정보를 조회할 수 있었다.

## ESC1 Exploitation

BloodHound를 통해 `_admin` 사용자가 `AWSJPDC0522` 호스트의 로컬 관리자 권한을 보유하고 있는 것을 확인하였다:

![Shibuya](/assets/htb-windows/shibuya/admin-bloodhound.png)

따라서 `_admin@shibuya.vl` UPN을 지정하여 인증서 발급을 시도하였다:


해당 오류는 인증서 템플릿에서 요구하는 최소 RSA 키 길이를 충족하지 못했기 때문에 발생한 것이다.
