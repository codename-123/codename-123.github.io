![Eureka](/assets/htb-linux/eureka/eureka.png)

**Eureka**는 어려운 난이도의 Linux 머신으로, Spring Boot 기반 Furni 애플리케이션의 `/actuator/heapdump` 엔드포인트 노출 취약점을 이용해 **메모리 덤프에서 민감한 자격증명을 추출**하며 초기 접근을 획득한다.  

이후 SSH로 시스템에 접속한 뒤, Eureka에 등록된 마이크로서비스 구조를 분석하여 Spring Cloud Gateway를 통해 `user-management-service` 에 인증을 위임하는 구조임을 파악한다. 공격자는 **Eureka의 안전하지 않은 서비스 등록 설정을 악용해 악성 `USER-MANAGEMENT-SERVICE` 를 등록하고, 게이트웨이가 해당 서비스로 로그인 요청을 라우팅하도록 유도**하여 정상 사용자 자격증명을 탈취한다.  

마지막으로 root 권한으로 실행되는 로그 분석 스크립트가 HTTP 상태 코드를 안전하지 않게 파싱하는 점을 이용해 `application.log` 에 조작된 페이로드를 삽입하고, 이를 통해 임의 명령 실행을 수행하여 최종적으로 루트 권한을 탈취한다.

# Exploitation

## Sensitive Information Extraction from Heapdump

[Spring Boot의 Heapdump 파일](https://www.wiz.io/blog/spring-boot-actuator-misconfigurations#1-exposed-heapdump-file-16)은 애플리케이션이 실행되는 순간의 모든 메모리 상태를 기록한 바이너리 스냅샷이다. 

본래 디버깅을 위한 용도이지만, 이 스냅샷 내에는 실행 중인 애플리케이션이 처리하던 **암호, 토큰, 클라우드 서비스 키, 사용자 자격 증명** 등 매우 민감한 정보들이 평문으로 포함될 수 있다.

`heapdump` 파일을 다운로드 받은 후, `strings` 명령어를 활용해 파일을 분석한 결과:

```bash
$ strings heapdump | grep -i "password"

# SKIP

{password=0sc@r190_S0l!dP@sswd, user=oscar190}!

# SKIP
```

위 처럼 `oscar190` 사용자의 자격증명이 발견되었다.

해당 정보의 정확한 용도를 파악하기 위해 해당 문자열의 전후 맥락을 분석해보니, **로컬 환경에서 동작 중인 MySQL 데이터베이스 접속을 위한 인증 정보**임을 확인할 수 있었다:

# Privilege Escalation

## Lateral Movement: oscar190 → miranda-wise

### Netstat Enumeration

셸 접근 후, 시스템에서 열려 있는 포트를 확인하였다:

```bash
oscar190@eureka:~$ netstat -antp

# SKIP

tcp6       0      0 127.0.0.1:8080          :::*                    LISTEN      -                   
tcp6       0      0 :::80                   :::*                    LISTEN      -                   
tcp6       0      0 127.0.0.1:8081          :::*                    LISTEN      -                   
tcp6       0      0 127.0.0.1:8082          :::*                    LISTEN      -                   
tcp6       0      0 :::22                   :::*                    LISTEN      -                   
tcp6       0      0 :::8761                 :::*                    LISTEN      - 
```

이처럼 내부망엔 `8080`, `8081`, `8082`, `8761` 포트가 활성화되어 있는 것을 확인하였다.

이 중 `8761` 포트는 [Spring Cloud Netflix Eureka](https://cloud.spring.io/spring-cloud-netflix/reference/html/#service-discovery-eureka-clients) 의 기본 포트로, 해당 시스템이 마이크로서비스들의 위치를 관리하는 **Service Discovery 기능**을 수행하고 있음을 알 수 있다.

### Service Enumeration

추가로 서버 내부를 탐색하던 중 `/var/www/web` 디렉토리에서 다음과 같은 구조를 확인할 수 있었다:

```bash
oscar190@eureka:/var/www/web$ ls -l

drwxrwxr-x 6 www-data developers 4096 Mar 18  2025 cloud-gateway
drwxrwxr-x 5 www-data developers 4096 Aug  5  2024 Eureka-Server
drwxrwxr-x 5 www-data developers 4096 Aug  5  2024 Furni
drwxrwxr-x 6 www-data developers 4096 Jul 23  2024 static
drwxrwxr-x 6 www-data developers 4096 Mar 19  2025 user-management-service
```

특히 `cloud-gateway` 디렉토리를 확인할 수 있었는데, 이는 **[Spring Cloud Gateway](https://cloud.spring.io/spring-cloud-gateway/reference/html/)**로 추정된다.

**Spring Cloud Gateway** 는 마이크로서비스 환경에서 **API Gateway 역할을 수행**하며, 클라이언트 요청을 적절한 내부 서비스로 라우팅하는 기능을 제공한다.

또한 **Spring Cloud Gateway** 에서는 라우팅 규칙을 정의하는 `application.yml` 설정 파일이 존재한다.

해당 파일에는 **Gateway** 가 클라이언트 요청을 어떤 내부 서비스로 전달할지에 대한 경로 설정이 정의되어 있다.

### Netflix Eureka SSRF

`cloud-gateway` 디렉토리 내부를 확인하던 중 `application.yaml` 설정 파일을 발견하였다.  
파일의 내용은 다음과 같다:

```bash
oscar190@eureka:/var/www/web/cloud-gateway/src/main/resources$ cat application.yaml

eureka:
  instance:
    hostname: localhost
    prefer-ip-address: false
  client:
    registry-fetch-interval-seconds: 20
    service-url:
      defaultZone: http://EurekaSrvr:0scarPWDisTheB3st@localhost:8761/eureka/

spring:
  cloud:
    client:
      hostname: localhost
    gateway:
      routes:
        - id: user-management-service
          uri: lb://USER-MANAGEMENT-SERVICE
          predicates:
            - Path=/login,/logout,/register,/process_register
        - id: furni
          uri: lb://FURNI
          predicates:
            - Path=/**

  application:
    name: app-gateway

server:
  port: 8080
  address: 127.0.0.1

management:
  tracing:
    sampling:
      probability: 1

logging:
  level:
    root: INFO
  file:
    name: log/application.log
    path: ./
```

> 이 취약점을 공략하기 위해 Backbase Engineering의 [Hacking Netflix Eureka](https://engineering.backbase.com/2023/05/16/hacking-netflix-eureka) 글을 참고하였다.

설정을 보면 **Gateway** 는 `/login`, `/logout`, `/register` 등의 인증 관련 요청을 `USER-MANAGEMENT-SERVICE` 로 라우팅하도록 구성되어 있다.

또한 `eureka.client.service-url.defaultZone` 설정을 통해 `http://EurekaSrvr:0scarPWDisTheB3st@localhost:8761/eureka/` 와 같이 Eureka 서버에 접근하기 위한 자격증명이 평문으로 포함되어 있는 것을 확인할 수 있었다.

우선 `logging.file.name` 설정을 통해 생성되는 `application.log` 파일을 확인해보았다.

특히 `/login` 요청이 다음과 같이 로그에 남는 것을 확인할 수 있다:

```log
2025-04-09T11:27:02.029Z  INFO 1234 --- [app-gateway] [reactor-http-epoll-4] c.eureka.gateway.Config.LoggingFilter    : HTTP POST /login - Status: 403
2025-04-09T11:27:02.161Z  INFO 1234 --- [app-gateway] [reactor-http-epoll-1] c.eureka.gateway.Config.LoggingFilter    : HTTP POST /login - Status: 403
2025-04-09T11:27:02.234Z  INFO 1234 --- [app-gateway] [reactor-http-epoll-2] c.eureka.gateway.Config.LoggingFilter    : HTTP POST /login - Status: 403
2025-04-09T11:27:02.286Z  INFO 1234 --- [app-gateway] [reactor-http-epoll-3] c.eureka.gateway.Config.LoggingFilter    : HTTP POST /login - Status: 403
```

위 로그를 통해 Gateway가 `/login` 요청을 `USER-MANAGEMENT-SERVICE` 로 라우팅하고 있음을 확인할 수 있다.

또한 `application.yaml` 파일에 포함된 Eureka 자격증명을 이용하면 Eureka 서버에 인증된 요청을 보낼 수 있으며, 이를 통해 새로운 서비스를 등록할 수 있는 권한이 존재함을 알 수 있다.  

따라서 공격자가 Eureka에 `USER-MANAGEMENT-SERVICE` 라는 이름의 악성 서비스를 등록할 경우, Gateway는 악성 서비스로 `/login` 요청을 라우팅하게 된다. 

이때 공격자가 자신의 서버를 해당 서비스로 등록하면, 로그인 요청이 공격자의 서버로 전달되게 된다.

이를 통해 공격자는 `nc` 등을 이용하여 피해자가 입력한 로그인 자격증명을 가로챌 수 있다. 

위 정보를 토대로 앞서 확인한 `user-management-service` 경로의 로그를 분석해 보았다:

```log
2025-04-09T11:39:01.878Z  INFO 1172 --- [USER-MANAGEMENT-SERVICE] [http-nio-127.0.0.1-8081-exec-1] c.e.Furni.Security.LoginSuccessLogger    : User 'miranda.wise@furni.htb' logged in successfully
2025-04-09T11:40:01.878Z  INFO 1172 --- [USER-MANAGEMENT-SERVICE] [http-nio-127.0.0.1-8081-exec-1] c.e.Furni.Security.LoginSuccessLogger    : User 'miranda.wise@furni.htb' logged in successfully
2025-04-09T11:41:01.878Z  INFO 1172 --- [USER-MANAGEMENT-SERVICE] [http-nio-127.0.0.1-8081-exec-1] c.e.Furni.Security.LoginSuccessLogger    : User 'miranda.wise@furni.htb' logged in successfully
2026-03-05T22:05:01.664Z  INFO 1328 --- [USER-MANAGEMENT-SERVICE] [http-nio-127.0.0.1-8081-exec-9] c.e.Furni.Security.LoginSuccessLogger    : User 'miranda.wise@furni.htb' logged in successfully
```

로그를 보면 `miranda.wise@furni.htb` 사용자의 로그인 성공 기록이 반복적으로 남아 있는 것을 확인할 수 있다.

또한 `2026-03-05 22:05:01` 시점에도 `miranda.wise@furni.htb` 계정의 로그인 성공 기록이 존재한다.
이를 통해 해당 계정이 현재도 사용 중인 **유효한 계정**임을 추정할 수 있다.

시간이 지난 후 동일한 로그를 다시 확인해 보았다:

```log
2025-04-09T11:40:01.878Z  INFO 1172 --- [USER-MANAGEMENT-SERVICE] [http-nio-127.0.0.1-8081-exec-1] c.e.Furni.Security.LoginSuccessLogger    : User 'miranda.wise@furni.htb' logged in successfully
2025-04-09T11:41:01.878Z  INFO 1172 --- [USER-MANAGEMENT-SERVICE] [http-nio-127.0.0.1-8081-exec-1] c.e.Furni.Security.LoginSuccessLogger    : User 'miranda.wise@furni.htb' logged in successfully
2026-03-05T22:11:01.622Z  INFO 1328 --- [USER-MANAGEMENT-SERVICE] [http-nio-127.0.0.1-8081-exec-5] c.e.Furni.Security.LoginSuccessLogger    : User 'miranda.wise@furni.htb' logged in successfully
```

위 로그인 `2026-03-05 22:11:01` 시점에서 또 `miranda.wise@furni.htb` 계정의 로그인 성공 기록이 존재한다.

이러한 패턴을 통해 해당 로그인 과정이 자동화된 작업에 의해 수행되고 있을 가능성을 고려할 수 있다.

해당 계정의 로그인 동작을 조금 더 분석하기 위해 [pspy](https://github.com/DominicBreuker/pspy) 도구를 활용하여 시스템에서 실행되는 프로세스를 관찰하였다.

pspy를 통해 `/opt/scripts` 경로의 `miranda-Login-Simulator.sh` 스크립트가 주기적으로 실행되고 있는 것을 확인할 수 있었으며, 로그를 보면 해당 스크립트가 `curl` 명령을 이용해 `http://furni.htb/login` 엔드포인트로 로그인 요청을 보내고 있음을 알 수 있다:

위 코드는 `application.log` 파일에서 `HTTP Status` 코드를 분석하는 기능을 수행한다.

먼저 `grep "HTTP.*Status: "` 를 통해 `HTTP Status` 로그 라인을 추출한 뒤, `grep -oP 'Status: \K.*'` 정규식을 사용하여 `Status:` 뒤에 오는 값을 추출한다.

예를 들어 다음과 같은 로그가 있을 경우:

```yaml
HTTP POST /login - Status: 403
```

`403` 값이 `code` 변수에 저장된다.

이후 스크립트는 다음과 같이 `-eq` 연산자를 사용하여 두 값을 비교한다:

```bash
[[ "$existing_code" -eq "$code" ]]
```

여기서 `-eq` 연산자는 산술 비교를 수행한다, 따라서 Bash는 `$code` 값을 단순 문자열이 아닌 산술식으로 해석하려고 시도한다.

문제는 `$code` 값이 로그 파일에서 직접 추출된 값이며 별도의 검증이 수행되지 않는다는 점이다.

따라서 공격자가 로그 파일에 다음과 같은 값을 삽입할 수 있다면 **Bash는 이를 배열 참조 형태의 산술식으로 해석**하고, 인덱스 부분에 포함된 `$(whoami)` 와 같은 명령 치환 문구를 먼저 평가하게 되며, 이 과정에서 해당 명령이 즉시 실행되는 결과가 나타난다:

```bash
a[$(whoami)]
```
