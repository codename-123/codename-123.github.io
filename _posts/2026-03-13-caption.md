![Caption](/assets/htb-linux/caption/caption.png)

**Caption**은 Hard 난이도의 Linux 머신으로, 기본 자격 증명을 통해 GitBucket에 접근하여 commit 기록에서 웹 포털 로그인 정보를 획득하고, **Web Cache Deception(`WCD`)**과 **XSS**를 이용한 응답 캐시 오염을 통해 관리자 세션을 탈취한다. 

이후 **HTTP/2 Cleartext 터널(`H2C Smuggling`)**을 사용해 HAProxy의 접근 제어를 우회하여 내부 서비스에 접근하고, 해당 서비스의 **Path Traversal 취약점(`CVE-2023-37474`)**을 악용해 **SSH ECDSA 개인 키**를 읽어 초기 foothold를 획득한다. 

마지막으로 루트 권한으로 실행되는 Apache Thrift 서비스의 **Command Injection 취약점**을 이용해 임의 명령을 실행하고 최종적으로 **루트 권한을 탈취**한다.

### Caption-Portal Source Code Analysis

`Caption-Portal` 저장소 내부에는 `app`, `config`, `README.md` 파일이 존재하는 것을 확인할 수 있다:

![Caption](/assets/htb-linux/caption/caption-portal.png)

이 중 `config` 디렉토리를 확인해 보면 내부에 [haproxy](https://www.haproxy.org/), [varnish](https://vinyl-cache.org/) 디렉토리가 존재하며, 웹 인프라에서 프록시 및 캐싱과 관련된 설정 파일들이 포함되어 있는 것을 확인할 수 있다:

![Caption](/assets/htb-linux/caption/config.png)

우선 `haproxy.cfg` 파일을 확인하여 HAProxy의 동작 방식을 분석하였다:

```bash
global
        log /dev/log    local0
        log /dev/log    local1 notice
        chroot /var/lib/haproxy
        stats socket /run/haproxy/admin.sock mode 660 level admin expose-fd listeners
        stats timeout 30s
        user haproxy
        group haproxy
        daemon
  
        # Default SSL material locations
        ca-base /etc/ssl/certs
        crt-base /etc/ssl/private
  
        # See: https://ssl-config.mozilla.org/#server=haproxy&server-version=2.0.3&config=intermediate
        ssl-default-bind-ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384:ECDHE-ECDSA-CHACHA20-POLY1305:ECDHE-RSA-CHACHA20-POLY1305:DHE-RSA-AES128-GCM-SHA256:DHE-RSA-AES256-GCM-SHA384
        ssl-default-bind-ciphersuites TLS_AES_128_GCM_SHA256:TLS_AES_256_GCM_SHA384:TLS_CHACHA20_POLY1305_SHA256
        ssl-default-bind-options ssl-min-ver TLSv1.2 no-tls-tickets
  
defaults
        log     global
        mode    http
        option  httplog
        option  dontlognull
        timeout connect 5000
        timeout client  50000
        timeout server  50000
        errorfile 400 /etc/haproxy/errors/400.http
        errorfile 403 /etc/haproxy/errors/403.http
        errorfile 408 /etc/haproxy/errors/408.http
        errorfile 500 /etc/haproxy/errors/500.http
        errorfile 502 /etc/haproxy/errors/502.http
        errorfile 503 /etc/haproxy/errors/503.http
        errorfile 504 /etc/haproxy/errors/504.http
  
  
frontend http_front
    bind *:80
    default_backend http_back
    acl multi_slash path_reg -i ^/[/%]+
    http-request deny if multi_slash
    acl restricted_page path_beg,url_dec -i /logs
    acl restricted_page path_beg,url_dec -i /download
    http-request deny if restricted_page
    acl not_caption hdr_beg(host) -i caption.htb
    http-request redirect code 301 location http://caption.htb if !not_caption
  
backend http_back
    balance roundrobin
    server server1 127.0.0.1:6081 check
```

해당 설정을 살펴보면 `frontend http_front` 에서 `80` 포트로 들어오는 HTTP 요청을 처리하며, 기본적으로 `http_back` 백엔드(`127.0.0.1:6081`)로 전달되는 것을 확인할 수 있다.

또한 몇가지 접근 규칙이 존재한다:

1. 정규표현식을 이용하여 URL 경로에 여러 개의 슬래시(`/`)가 포함된 요청을 차단한다.
2. `/logs`, `/download` 경로로 시작하는 요청을 URL 디코딩(`url_dec`) 후 검사하여 해당 경로에 대한 접근을 HAProxy 단계에서 차단한다.
3. `Host` 헤더가 `caption.htb`가 아닌 경우 `http://caption.htb`로 `301` 리다이렉트가 수행된다.

백엔드 주소(`127.0.0.1:6081`)의 동작을 확인하기 위해 `service` 디렉토리에 존재하는 `varnish.service` 파일을 확인하였다:

```bash
[Unit]
Description=Varnish Cache, a high-performance HTTP accelerator
Documentation=https://www.varnish-cache.org/docs/ man:varnishd
  
[Service]
Type=simple
  
# Maximum number of open files (for ulimit -n)
LimitNOFILE=131072
  
# Locked shared memory - should suffice to lock the shared memory log
# (varnishd -l argument)
# Default log size is 80MB vsl + 1M vsm + header -> 82MB
# unit is bytes
LimitMEMLOCK=85983232
ExecStart=/usr/sbin/varnishd \
          -j unix,user=vcache \
          -F \
          -a localhost:6081 \
          -T localhost:6082 \
          -f /etc/varnish/default.vcl \
          -S /etc/varnish/secret \
          -s malloc,256m \
          -p feature=+http2
ExecReload=/usr/share/varnish/varnishreload
ProtectSystem=full
ProtectHome=true
PrivateTmp=true
PrivateDevices=true
  
[Install]
WantedBy=multi-user.target
```

특히 주목할 부분은 `-a localhost:6081` 옵션이다.

해당 옵션은 **Varnish가 HTTP 요청을 수신하는 리스닝 주소**를 의미하며, `haproxy.cfg` 에서 확인된 백엔드 주소(`127.0.0.1:6081`)와 동일하다.

또한 `-p feature=+http2` 옵션을 통해 HTTP/2 기능이 활성화되어 있는 것을 확인할 수 있으며, 이는 이후 **H2C Smuggling 공격**과 관련될 가능성을 보여준다.

이를 통해 외부에서 들어오는 요청이 HAProxy를 거쳐 Varnish 캐시 서버로 전달되는 구조임을 확인할 수 있다:

```text
Client → HAProxy → Varnish → Backend
```

이후 `varnish` 디렉토리에 존재하는 `default.vcl` 파일을 분석하였다:

```vcl
vcl 4.0;
  
backend default {
    .host = "127.0.0.1";
    .port = "8000";
}
  
sub vcl_recv {
  // update for prod - CR-3045
}
  
sub vcl_backend_response {
  // update for prod - CR-3045
}
  
sub vcl_deliver {
  // update for prod - CR-3045
}
```

초기 `nmap` 열거 과정에서 웹 서버의 `Werkzeug` 헤더(`Werkzeug/3.0.1 Python/3.10.12`)가 확인되었다. 

`Werkzeug`는 Python 기반 WSGI 서버로, 일반적으로 Flask 프레임워크에서 사용되므로 백엔드에서 Flask 기반 웹 애플리케이션이 동작하고 있을 가능성을 추정할 수 있다.

## Bypassing HAProxy via H2C Smuggling

앞서 확인한 것처럼 `/logs` 경로에 대한 접근은 HAProxy의 ACL 규칙에 의해 프록시 레벨에서 차단되고 있다.

또한 `varnish.service` 설정 파일을 살펴보면 `-p feature=+http2` 옵션이 활성화되어 있어 Varnish가 `HTTP/2` 기능을 지원하고 있는 것을 확인할 수 있다.

이러한 구조를 고려했을 때, 프록시의 접근 제어를 우회하기 위해 **H2C Smuggling** 기법을 사용할 수 있다.

[h2c](https://bishopfox.com/blog/h2c-smuggling-request)는 HTTP/1.1 연결을 **HTTP/2 Cleartext(H2C)**로 업그레이드하여 리버스 프록시 뒤에 위치한 내부 서비스와 직접 통신할 수 있도록 하는 기법이다.

GitHub에 공개된 [h2csmuggler](https://github.com/BishopFox/h2csmuggler) 도구를 활용하여 `/logs` 경로에 접근을 시도하였다:

```bash
$ python3 h2csmuggler.py -x http://caption.htb http://caption.htb/logs

<!DOCTYPE html>
<html lang="en">

<head>
  <meta charset="UTF-8">
  
    <script src="https://cpwebassets.codepen.io/assets/common/stopExecutionOnTimeout-2c7831bb44f98c1391d6a4ffda0e1fd302503391ca806e7fcc7b9b87197aec26.js"></script>


  <title>Caption Portal Login</title>

  <link rel="canonical" href="https://codepen.io/Tushar-Sandhu/pen/YzRROwd">
  <script>
    window.console = window.console || function(t) {};
  </script>

# SKIP
</body>

</html>

[INFO] Requesting - /logs
:status: 302
server: Werkzeug/3.0.1 Python/3.10.12
date: Fri, 13 Mar 2026 12:38:24 GMT
content-type: text/html; charset=utf-8
content-length: 189
location: /
x-varnish: 262198
age: 0
via: 1.1 varnish (Varnish/6.6)
x-cache: MISS

<!doctype html>
<html lang=en>
<title>Redirecting...</title>
<h1>Redirecting...</h1>
<p>You should be redirected automatically to the target URL: <a href="/">/</a>. If not, click the link.
```

기존에는 `/logs` 경로에 접근할 경우 `403 Forbidden` 응답이 반환되었지만, **H2C Smuggling**을 통해 요청을 전달했을 때 `302 Redirect` 응답이 반환되는 것을 확인할 수 있었다.

즉, H2C Smuggling을 통해 **HAProxy의 ACL을 우회**할 수 있음을 확인하였다.

## Web Cache Deception & Cache Poisoning (XSS)

Burp Suite를 이용하여 `/home` 경로에 대한 요청을 확인해보면, 응답 헤더에 Varnish 관련 캐시 헤더가 포함되어 있는 것을 확인할 수 있다:

![Caption](/assets/htb-linux/caption/home.png)

그러나 동일한 요청을 반복적으로 전송해보면 `age` 값이 계속 `0` 으로 유지되는 것을 확인할 수 있다.  

이는 해당 요청이 캐시에 저장되지 않고 매번 백엔드로 전달되고 있음을 의미하며, Varnish 캐시가 정상적으로 동작하지 않는 것으로 보인다.

하지만 `/firewalls` 경로에서는 응답 헤더의 `Cache-Control` 값이 `public, max-age=120`으로 설정되어 있는 것을 확인할 수 있다.  

또한 동일한 요청을 반복적으로 전송하면 `age` 값이 증가하는 것을 확인할 수 있으며, 이는 해당 페이지가 Varnish 캐시에 저장되고 있음을 의미한다:

![Caption](/assets/htb-linux/caption/firewall.png)

즉, 해당 요청은 최초에는 백엔드 서버로 전달되지만 이후 `120` 초 동안 캐시 서버에 저장되며, 동일한 요청이 들어올 경우 백엔드를 거치지 않고 캐시된 응답을 반환하게 된다.

이후 응답을 분석하던 중 JavaScript 파일을 로드하는 `<script>` 태그에서 다음과 같은 URL을 확인할 수 있었다:

![Caption](/assets/htb-linux/caption/javascript.png)

해당 URL의 `utm_source` 파라미터에 내부 프록시 주소(`internal-proxy.local`)가 포함되어 있는 것을 확인할 수 있다.

이를 통해 애플리케이션이 프록시 환경과 관련된 정보를 응답 페이지에 반영하고 있을 가능성을 확인할 수 있다.

따라서 해당 값이 HTTP 헤더를 통해 조작 가능한지 확인하기 위해 여러 프록시 관련 헤더를 조작하여 테스트를 진행하였다:

![Caption](/assets/htb-linux/caption/test.png)

그 결과 `X-Forwarded-Host` 헤더 값이 응답 페이지의 `utm_source` 파라미터에 그대로 반영되는 것을 확인할 수 있었다.

`X-Forwarded-Host` 헤더에 다음과 같은 XSS payload를 삽입하여 테스트를 진행하였다:

```http
X-Forwarded-Host: test"><script>alert(1)</script>
```

그 결과 payload가 응답 페이지에 그대로 반영되며, XSS가 정상적으로 발생하는 것을 확인할 수 있었다:

![Caption](/assets/htb-linux/caption/xss.png)

현재 `/firewalls` 페이지는 `Cache-Control: public, max-age=120` 설정으로 인해 약 120초 동안 캐시에 저장된다. 

또한 앞서 확인한 것처럼 `X-Forwarded-Host` 헤더 값을 통해 XSS 공격이 가능하다.  

따라서 해당 페이지에 XSS payload를 삽입한 뒤 캐시에 저장되도록 유도하면 **Cache Poisoning 공격**이 가능하다는 것을 확인할 수 있다.

개발자 도구를 통해 쿠키 속성을 확인해 보면 `session` 쿠키에 `HttpOnly` 옵션이 설정되어 있지 않은 것을 확인할 수 있다:

![Caption](/assets/htb-linux/caption/httponly.png)

따라서 JavaScript를 이용하여 사용자의 쿠키를 탈취할 수 있다.

우선 로컬 터미널에서 HTTP 서버를 실행하였다:

```bash
$ python3 -m http.server
```

## Path Traversal (CVE-2023-37474)

JWT는 `header.payload.signature` 구조로 이루어져 있으며, 이 중 `payload` 부분에는 사용자와 관련된 클레임 정보가 포함되어 있다.

payload는 Base64로 인코딩되어 있기 때문에 디코딩을 통해 내용을 확인할 수 있으며, 디코딩 결과 `username` 값이 `admin` 으로 설정되어 있는 것을 확인할 수 있다:

```bash
$ echo "eyJ1c2VybmFtZSI6ImFkbWluIiwiZXhwIjoxNzczNDIzMTQ0fQ" | base64 -d

{"username":"admin","exp":1773423144}
```
